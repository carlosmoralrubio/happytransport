HappyTransport Carrier Load Sales Automation

Proof of Concept Presentation for Inbound Carrier Calls

Date: 8 March 2026

1. Executive Summary

This proof of concept demonstrates a production-oriented workflow for automating inbound carrier load sales using HappyRobot plus a backend decision API and analytics dashboard. The solution helps your brokerage team respond faster, keep interactions consistent, and capture every call outcome for continuous improvement.

The POC is already functional end-to-end for load retrieval, outcome capture, and reporting. It is designed so that your operations team can start with CSV data and evolve to enterprise-grade infrastructure with minimal rework.

2. Customer Problem We Are Solving

  • Inbound carrier calls require fast load matching under time pressure.
  • Manual processes lead to inconsistent qualification and booking outcomes.
  • Teams often lack structured call-level data to improve conversion and margin.
  • Without feedback loops, negotiation performance and carrier quality are hard to optimize.

4. High-Level Solution Design

  1. Carrier calls inbound to HappyRobot voice agent.
  2. Agent extracts intent and shipment constraints from the conversation.
  3. Backend API returns matching loads in near-real time.
  4. Agent offers options and handles negotiation logic.
  5. At call end, agent posts structured outcome metrics for analytics.
  6. Dashboard displays booking rate, negotiation impact, and sentiment trends.

5. Technical Deep Dive (Backend API)

5.1 Load Matching Endpoint

GET /v1/loads

  • Combines optional filters in a single call (origin, destination, equipment, weight range, pickup/delivery windows).
  • Uses case-insensitive partial matching to handle natural language variations from voice calls.
  • Converts date filters reliably with pandas datetime parsing.
  • Returns both total_matches and filters_applied for transparency.

5.2 Metrics Capture Endpoint

POST /v1/metrics

  • Captures booking outcomes including negotiation context, sentiment, duration, and carrier validation details.
  • Enforces controlled values for sentiment and outcome via schema validation.
  • Records UTC timestamps for clean historical analysis.
  • Uses thread locking for safe concurrent writes when multiple agent sessions submit metrics simultaneously.

5.3 Metrics Retrieval Endpoint

GET /v1/metrics

  • Supports filtering by outcome, sentiment, and carrier MC number.
  • Includes limit/offset pagination for dashboard and reporting use cases.
  • Returns total_records plus current page results for stable UI behavior.

5.4 Security and Reliability

  • Authentication through X-API-Key with 401/403 handling.
  • CORS configured for production frontend and local development.
  • Health endpoint validates dataset availability for uptime checks.
  • Environment-driven configuration for dataset path, metrics path, and API keys.

6. Analytics and Operations View

The React dashboard currently provides operational and quality KPIs such as:

  • Total processed interactions.
  • Booking rate (booked outcomes over total).
  • Average price difference vs. loadboard rate.
  • Average call duration.
  • Valid MC ratio.
  • Outcome and sentiment distributions.
  • Trend line for price delta over time.

If the API is unavailable, the dashboard can fall back to demo data to keep stakeholder demonstrations uninterrupted.

10. Implementation Notes and Next Phase

Current POC Assumptions

  • Load and metrics persistence are CSV-based for speed of implementation.
  • Single-service architecture is optimized for rapid validation and low operating complexity.

Recommended Next Steps for Production

  • Move metrics storage from CSV to PostgreSQL for auditability and scale.
  • Add role-based access and key rotation policy for enterprise security.
  • Introduce richer matching and ranking logic (margin, capacity fit, historical acceptance).
  • Add SLA monitoring and alerting (latency, error rate, booking drop-offs).
  • Integrate carrier/compliance systems for deeper automated qualification.
